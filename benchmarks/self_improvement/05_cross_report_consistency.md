# Spec 05: Cross-Report Data Consistency

**Scope:** Verify that numbers, page counts, tool names, and rankings are
consistent across all benchmark reports and their source data.

**When to run:** After any benchmark re-run, after updating reports, or when
a report "doesn't smell right."

**Automated checks:** The most critical subset of this spec (README vs source
reports) is now automated by `check_cross_report_consistency.py`. Run it as
part of the pre-commit gate. The manual checks below cover areas the script
does not (page counts vs raw JSONL, tool name casing, aggregation math,
checkpoint staleness).

---

## Why this matters

Each report is generated by a different script from overlapping source data.
When one script is re-run but others aren't, numbers drift apart. This spec
catches those inconsistencies before readers do.

Common failure modes:
- Stale retrieval checkpoints cause RETRIEVAL_COMPARISON.md to show old page counts
- SPEED_COMPARISON.md updated after a re-crawl but COST_AT_SCALE.md still uses old chunk ratios
- A tool name is spelled differently across reports (e.g., "colly+md" vs "colly")

---

## What to check

### 1. Page counts agree across reports

The same tool+site combination should show the same page count in every report
that references it. Pull page counts from each report and compare.

```bash
# Extract page counts from SPEED_COMPARISON.md
echo "=== SPEED ==="
grep -E "^\| (markcrawl|crawl4ai|scrapy|crawlee|colly|playwright|firecrawl)" benchmarks/SPEED_COMPARISON.md | head -20

# Extract from RETRIEVAL_COMPARISON.md
echo "=== RETRIEVAL ==="
grep -E "^\| (markcrawl|crawl4ai|scrapy|crawlee|colly|playwright|firecrawl)" benchmarks/RETRIEVAL_COMPARISON.md | head -20

# Compare against actual JSONL line counts
echo "=== ACTUAL DATA ==="
for dir in benchmarks/runs/run_*/*/; do
  tool=$(basename $(dirname "$dir"))
  site=$(basename "$dir")
  count=$(wc -l < "$dir/pages.jsonl" 2>/dev/null || echo "MISSING")
  echo "$tool/$site: $count"
done
```

### 2. Tool names are consistent

All reports must use identical tool names. The canonical names are:

| Tool name | Variants to catch |
|-----------|-------------------|
| markcrawl | MarkCrawl, mark_crawl |
| crawl4ai | Crawl4AI, crawl4ai-batch |
| crawl4ai-raw | crawl4ai_raw |
| scrapy+md | scrapy, scrapy-md |
| crawlee | Crawlee |
| colly+md | colly, colly-md |
| playwright | Playwright |
| firecrawl | Firecrawl, FireCrawl |

```bash
# Check for inconsistent tool name casing/spelling in tables
grep -Pn "(?i)(markcrawl|crawl4ai|scrapy|crawlee|colly|playwright|firecrawl)" benchmarks/*.md | \
  grep -v "^benchmarks/METHODOLOGY\|^benchmarks/SELF_IMPROVEMENT\|self_improvement" | \
  grep "^\|" | sort | head -40
```

### 3. Rankings are consistent

If SPEED says scrapy+md is fastest and COST says it's cheapest-per-page,
those claims should not contradict each other. Check:

- [ ] SPEED_COMPARISON.md: which tool is ranked #1 overall?
- [ ] COST_AT_SCALE.md: do cost rankings align with chunk count rankings?
- [ ] ANSWER_QUALITY.md: does the #1 tool match what COST says about quality?

### 4. Chunk counts are consistent

RETRIEVAL_COMPARISON.md and COST_AT_SCALE.md both reference chunk counts.
They must agree.

```bash
# Chunk counts from RETRIEVAL
grep -A2 "chunks" benchmarks/RETRIEVAL_COMPARISON.md | head -20
# Chunk counts from COST
grep -A2 "chunks" benchmarks/COST_AT_SCALE.md | head -20
```

### 5. Overall summary numbers are correct

Each report has a summary table with aggregated numbers. Verify the
aggregation is correct:

- [ ] SPEED: total pages = sum of per-site pages
- [ ] SPEED: avg pages/sec = total pages / total time (not mean of per-site rates)
- [ ] RETRIEVAL: overall hit rate = weighted by query count per site (not simple mean)
- [ ] ANSWER_QUALITY: overall score = mean of per-site scores

### 6. Stale checkpoint detection

Checkpoints cache computed results. If source data changes but checkpoints
don't, reports show stale numbers.

```bash
# Compare checkpoint timestamps to source data timestamps
echo "=== Retrieval checkpoints ==="
ls -lt benchmarks/retrieval_checkpoints/ | head -10
echo ""
echo "=== Source JSONL files ==="
ls -lt benchmarks/runs/run_*/*/pages.jsonl | head -10
echo ""
echo "=== Answer quality checkpoints ==="
ls -lt benchmarks/answer_quality_checkpoints/ | head -10
```

**Rule:** If any checkpoint file is older than its source `pages.jsonl`,
flag it as a finding. Do NOT re-run benchmark scripts during a review —
they take hours and consume API credits. Instead, note which checkpoints
are stale so the user can schedule a re-run.

### 7. Generation timestamps

Each report has a "Generated:" timestamp. Check that all reports were
generated from the same run.

```bash
grep -n "Generated:\|generated:" benchmarks/*.md
```

If timestamps differ significantly, some reports may be using different
source data.

### 8. Cross-reference accuracy

When one report cites a number from another report, verify the number matches.

Common cross-references:
- COST_AT_SCALE cites chunk ratios from RETRIEVAL_COMPARISON
- ANSWER_QUALITY cites retrieval hit rates from RETRIEVAL_COMPARISON
- SPEED_COMPARISON cites quality findings from QUALITY_COMPARISON

```bash
# Find cross-reference claims
grep -n "see \[" benchmarks/*.md | grep -v self_improvement
```

---

## What "good" looks like

- Page counts for every tool+site match across all reports
- Tool names spelled identically everywhere
- Summary aggregations are mathematically correct
- No checkpoint is older than its source data
- All reports generated from the same run (within a few hours)
- Cross-referenced numbers match their source reports
