**Scope:** All benchmark Markdown reports checked against the style guide
defined in `/CLAUDE.md`.

**When to run:** After any report is regenerated, after style guide updates,
or before committing report changes.

## What to check

### 1. Automated checks (lint_reports.py)

Run the linter first -- it catches the mechanical issues.

```bash
python benchmarks/lint_reports.py
```

The linter checks:
- [ ] Style version tag (`<!-- style: v2, YYYY-MM-DD -->`) present after title
- [ ] One-line answer (states finding, not "this report measures...")
- [ ] Cross-references to related reports
- [ ] All tools included in summary tables
- [ ] MarkCrawl row is bold (`**`)
- [ ] No emojis
- [ ] Link to METHODOLOGY.md

If the linter fails, fix the issues before proceeding to manual checks.

### 2. Structure (manual check per report)

Each report should follow this order:

1. **Title** -- `# Report Name`
2. **Style version tag** -- `<!-- style: v2, YYYY-MM-DD -->`
3. **One-line answer** -- states the finding, not what was measured
4. **Context for the numbers** -- explains metrics in plain language
5. **Summary table** -- all tools, ranked by primary metric
6. **Narrative interpretation** -- what do results mean, honest trade-offs
7. **Per-site breakdowns** -- detailed results
8. **Methodology** -- how to reproduce

- [ ] Is the one-line answer a finding, not a description?
  - Good: "Scrapy+md is the fastest tool overall (9.1 pages/sec)"
  - Bad: "This benchmark compares crawl speed across 8 tools"
- [ ] Are metrics explained before the first table?
- [ ] Does the summary table include all 8 tools?
- [ ] Is the summary table sorted by primary metric (best first)?
- [ ] Is markcrawl's row bold? Is it in the correct rank position?
- [ ] Does the narrative acknowledge when markcrawl loses?

### 3. Formatting rules

- [ ] **Percentages** in "vs markcrawl" columns (not multipliers)
- [ ] **Consistent units** within each sentence (don't mix "1.21x" and "1.3%")
- [ ] **Rounding:** percentages to 1 decimal, dollars to 2 decimals below $1 / no decimals above $10, scores to 2 decimals
- [ ] **No emojis** anywhere in reports
- [ ] **Cost tables** state pricing assumptions inline or link to methodology

```bash
# Check for emojis (common Unicode emoji ranges)
grep -Pn '[\x{1F600}-\x{1F64F}\x{1F300}-\x{1F5FF}\x{1F680}-\x{1F6FF}\x{2600}-\x{26FF}]' benchmarks/*.md || echo "No emojis found"
# Check for multiplier format in vs-markcrawl columns
grep -n "vs markcrawl" benchmarks/*.md
```

### 4. Cross-references

Each report should link to related reports:

| Report | Must link to |
|--------|-------------|
| SPEED_COMPARISON | QUALITY_COMPARISON, METHODOLOGY |
| QUALITY_COMPARISON | RETRIEVAL_COMPARISON, METHODOLOGY |
| RETRIEVAL_COMPARISON | ANSWER_QUALITY, METHODOLOGY |
| ANSWER_QUALITY | COST_AT_SCALE, RETRIEVAL_COMPARISON, METHODOLOGY |
| COST_AT_SCALE | ANSWER_QUALITY, RETRIEVAL_COMPARISON, METHODOLOGY |

```bash
# Check cross-references exist
for report in SPEED_COMPARISON QUALITY_COMPARISON RETRIEVAL_COMPARISON ANSWER_QUALITY COST_AT_SCALE; do
  echo "=== $report ==="
  grep -c "METHODOLOGY\|\.md)" "benchmarks/${report}.md"
done
```

### 5. Tone and credibility

- [ ] Does the report acknowledge when markcrawl loses on a metric?
- [ ] Are small differences (<2% on a 5-point scale) described as "similar"?
- [ ] Are comparisons made against the strongest competitor, not the weakest?
- [ ] Does it say "all tools" when comparing, not cherry-pick?

### 6. Page-specific checks

**SPEED_COMPARISON.md:**
- [ ] Notes which tools use JS rendering vs plain HTTP
- [ ] Shows median + std dev across iterations
- [ ] Notes page completeness (tools that missed pages)

**QUALITY_COMPARISON.md:**
- [ ] Shows content signal %, preamble words, and repeat rate together
- [ ] Explains why preamble/repeat rate matter for RAG

**RETRIEVAL_COMPARISON.md:**
- [ ] Shows all 4 retrieval modes
- [ ] Includes raw hit counts alongside percentages
- [ ] Has "best mode per tool" digest before the full 28-row table

**ANSWER_QUALITY.md:**
- [ ] Shows all 4 scoring dimensions + overall average
- [ ] Explains the scoring scale before the summary table
- [ ] Per-query breakdowns in collapsible `<details>` sections

**COST_AT_SCALE.md:**
- [ ] Separates storage costs from query costs
- [ ] Shows named scenarios (small, mid, large)
- [ ] Includes formulas and pricing source links

### 7. README sync

The README has a benchmark summary in the `<details>` section. Check it
matches current report findings.

```bash
# Check README benchmark section exists and mentions key metrics
grep -A 5 "How it compares" README.md
```

## What "good" looks like

- `python benchmarks/lint_reports.py` passes with zero errors
- Every report has the correct structure (title, answer, context, table, narrative)
- All summary tables include all 8 tools, sorted by primary metric
- Cross-references are complete and point to existing files
- Tone is honest -- losses acknowledged, small gaps not overclaimed
- README summary matches current report findings
